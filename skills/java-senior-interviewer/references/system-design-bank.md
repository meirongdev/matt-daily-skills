# System Design Bank (系统设计题库)

Use in the `[30–40m] 系统设计` block. **Pick one scenario** whose topic the candidate has claimed on the resume — that forces them to deliver real experience rather than recite textbook patterns.

## How to run the block

Give the candidate 2 minutes to clarify requirements. **A strong candidate will clarify before designing.** If they dive into "使用 Redis + MQ..." before asking scale / consistency / latency questions, that's a calibration signal — push back with "先问我你需要知道什么".

Evaluate across these dimensions:

| 维度 | Engineer 及格 | Senior 及格 | Senior 优秀 |
|---|---|---|---|
| 需求澄清 | 能问几个关键问题 | 主动问容量 + 一致性 + 延迟要求 | 把隐含约束挖出来 (e.g., "资金是否涉及？跨境？监管审计？") |
| 容量估算 | 能做粗估 | 数量级合理、写速率 / 存储 / 带宽 都覆盖 | 能从容量反推架构选择 |
| 数据模型 | 表结构正确 | 索引、分片键、冷热分离合理 | 预留演进空间、schema 版本控制 |
| 写路径 | 能走通 | 处理幂等、事务 | 命名一致性方案及其 trade-off |
| 读路径 | 能走通 | 分层缓存、索引策略 | 讨论读写分离的延迟影响 |
| 失败模式 | 至少提 1 个 | 至少 3 个，并给出应对 | 主动设计 degraded mode |

## 评分模板

- 🟢 **优秀**：命中「优秀」≥ 3 个维度。可直接下 hire bar。
- 🟡 **及格**：达到对应级别的及格线。
- 🔴 **不过**：关键维度缺失（比如秒杀场景完全不提限流，或支付对账完全不提幂等）。

## 场景 1 — 支付对账系统

### 题面

> "你们公司每天通过多个第三方支付渠道（支付宝、微信、银联、Apple Pay）收款 100 万笔，单笔金额 1~10 万元。现在要你设计一个对账系统：每天 T+1 下载各渠道的交易流水，和我们本地的订单表核对，找出所有差异（我方有他们没 / 他们有我方没 / 金额不一致 / 状态不一致）。差异出来后进入人工处理流程。"

### 考察重点

- **对大文件 / 大批量数据处理的直觉**（百万级 × 多渠道 = 可能几百万条 / 天）。
- **对账的核心是"找差集"——怎么高效？**
- **差错处理 / 补偿 / 幂等 / 流程化的思维**（对账是否可以重跑？跑到一半挂了怎么办？）。
- **金融业务的敏感度**（数据留痕、审计、对账完成前不允许关账）。

### 需要澄清的问题（优秀候选人会主动问）

- 渠道流水文件大小和格式？CSV / XML / API？有标准吗？
- T+1 是从渠道拿到文件的时间，还是我们完成对账的时间？SLA 是几小时？
- 差异量级估计？（正常是万分之一以下；如果高说明系统有问题，对账只是发现手段）
- 历史数据要保留多久？（金融监管通常 7 年以上）
- 谁来看结果？是财务日清，还是 ops 值守？
- 是否需要支持 T+N 回溯对账（某天发现历史数据有问题要重新比对）？

### 合格方案轮廓

**数据流**：
1. **文件下载**：每渠道独立 Job，失败重试，下载完成入库（文件元数据表 + 文件本体放对象存储）。
2. **解析入库**：把渠道流水规范化到统一 schema（渠道、交易号、我方订单号、金额、时间、状态），批量写入临时表（当天分区）。
3. **比对**：SQL `FULL OUTER JOIN` 本地订单 × 渠道流水，按 `外部订单号 + 金额 + 状态` 匹配。输出四类差异：
   - 本地有渠道无（可能我方多扣 / 重复 / 渠道未到账）
   - 渠道有本地无（可能我方丢单 / 渠道重复）
   - 金额不一致（可能数据错误 / 汇率 / 手续费）
   - 状态不一致（我方已完成但渠道失败 / 反之）
4. **差异入库**：差异表，带处理状态（pending / assigned / resolved），推送到人工处理。
5. **审计日志**：对账任务的开始/结束时间、数据量、差异量、操作人。

**关键技术点**：
- 海量 JOIN 的性能：按日期分区 + 按订单号/渠道建索引；或用列存 / Spark / Flink batch。
- 幂等：对账 Job 可重跑（按日期 + 渠道幂等）；差异记录用 (日期, 渠道, 订单号) 唯一键避免重复。
- 分段提交 + 断点续跑：长 Job 分片处理，记录 checkpoint。

### 加分回答

- **双向对账**：不仅渠道 vs 本地，还本地 vs 内存账 / 本地 vs 结算账，形成三方对账。
- **实时预警**：除了 T+1 批量对账，加实时流（Kafka + Flink）做分钟级异常检测，重大差异不等 T+1。
- **数据质量**：渠道侧也可能给错文件（漏条、重复、格式错），要有文件级完整性校验（行数 + 金额合计 hash）。
- **监管视角**：差异处理要留痕，operator 操作不可删改，采用 append-only 日志。
- **数据安全**：渠道流水含敏感信息，落盘加密、按需脱敏、最小权限访问。

### 常见失分

- 只讲"用 MQ 异步处理"，没讲怎么比对。
- 不处理文件下载失败 / 部分失败。
- 不考虑重跑幂等。
- 不问金额精度（浮点 vs BigDecimal / 分 vs 元）—— 金融场景误差一分都是事故。

## 场景 2 — 高并发秒杀

### 题面

> "双 11 零点开售某款商品，库存 1 万件，预期 100 万用户同时点击购买按钮。要求：不能超卖、成功下单的用户能在 5 秒内拿到订单号，没抢到的用户明确知道没抢到；整个系统不能因为这 1 万件商品崩掉其他业务。"

### 考察重点

- **分层削峰**：前端限流 → 网关 → 业务服务 → DB。
- **不超卖的核心**：扣库存必须原子（Redis + Lua 或 DB 行锁）。
- **用户体验**：异步下单 vs. 同步下单的权衡。
- **隔离**：秒杀不能把交易主链路打垮。

### 需要澄清的问题

- 这 1 万件库存是仓储真实库存，还是营销活动独占的虚拟库存？
- 是否支持预约？（预约型秒杀可以把 100 万压力分散到预约期）
- 单用户限购几件？（影响幂等和风控）
- 是否要识别刷子？（黄牛 / 脚本）
- 超时未支付自动释放库存的逻辑？（一般 15 min）
- 对商家端的一致性要求（库存中心看到的数字是实时的吗）？

### 合格方案轮廓

**削峰分层**：
1. **前端**：点击按钮后立即置灰 + 随机延迟 0–1s 分散请求；静态资源 CDN；按钮动态激活（活动开始前按钮不可点）。
2. **网关**：按 user_id 限流（每用户 1 QPS），黑名单 IP / 设备指纹。
3. **服务层**：进一步限流 + 令牌桶 + 验证码（高价值活动）；对非会员 / 新注册用户再降权。
4. **库存**：Redis 原子扣减（Lua 脚本：check + decr），扣成功后发下单消息到 MQ，同步返回"正在下单"的 requestId。
5. **异步下单**：MQ 消费者创建订单、扣减 DB 库存、发支付链接。用户轮询 requestId 获取结果（5 秒内返回订单号或失败）。
6. **兜底**：DB 层库存表 `stock >= ?` WHERE + UPDATE 作为最后一道防线（防 Redis 异常超卖）。

**不超卖的要点**：
- Redis Lua 是唯一真相源；DB 是审计和兜底。
- Redis 到 DB 异步，即使最终 DB 数量 > Redis 成功数，也可以通过限库存补救。
- 超时未支付释放库存 → Redis incr（还要防止释放错 key）。

### 加分回答

- **预热**：活动开始前提前把库存加载到 Redis，避免冷启动 miss。
- **分段库存**：1 万件库存切 N 段分散到 N 个 key / 实例，避免单 key 热点（取模/一致性哈希）。Redis Cluster 下这一点很关键。
- **压测**：全链路压测，找到真实瓶颈。哪个下游先顶不住（数据库连接池？消息队列？第三方风控？）。
- **降级策略**：非核心功能可暂停（推荐、评价、日志详尽采样）。秒杀专用降级开关。
- **数据观测**：实时大盘监控库存消耗速度、失败率、平均响应时间、用户 Wait 时间。
- **公平性**：随机排队 vs. 先到先得的 trade-off；有些电商用预约+随机中奖更适合高价值稀缺品。

### 常见失分

- 一上来就说"用 Redis + MQ"，但说不清怎么防超卖。
- 不讲兜底，过度依赖 Redis（Redis 故障就超卖）。
- 不讲隔离，把秒杀和常规下单混到一个服务（雪崩风险）。
- 不估算容量：100 万用户涌入 = 多少 QPS？Redis 单实例能扛几万？你需要几个？

## 场景 3 — 日志采集系统

### 题面

> "公司有 1000 台业务机器，每台机器每天产生 10GB 日志（访问日志、应用日志、GC 日志）。要求：日志在 1 分钟内可在集中查询界面搜到，原文保留 30 天可下载，分析报表保留 1 年。日志不能丢（至少 99.99% 可达）。"

### 考察重点

- **大量数据的管道设计**：采集 → 传输 → 存储 → 查询。
- **可靠性 / 丢失 / 重复 的权衡**。
- **成本意识**：10TB/天 * 30 天 = 300TB，按原文存 S3 vs. 搜索引擎的成本差。
- **查询性能**：1000 台 × 30 天里查"某个 traceId"要在秒级返回。

### 需要澄清的问题

- 查询场景？（全文检索 / 按字段过滤 / 聚合分析）
- 高基数字段？（trace_id / user_id / device_id → 影响索引设计）
- 业务是否需要从日志反推告警？（日志和监控的边界）
- 敏感信息脱敏要求？（身份证 / 手机号 / 银行卡）
- 机房分布？跨 IDC 带宽预算？

### 合格方案轮廓

**管道**：
1. **采集端**：Filebeat / Fluent Bit 读取日志文件（tail + checkpoint），支持断点续传；本地磁盘作为 buffer 防采集端崩溃丢数据。
2. **传输**：Kafka 作为缓冲层；按 topic 分类（access / app / gc）；按业务拆分 topic 或用 key 路由到分区。
3. **存储与索引**：
   - **热数据（0–7 天）**：Elasticsearch / OpenSearch，全文索引 + 结构化字段。
   - **温数据（7–30 天）**：ES 冷节点（更便宜磁盘）或 ClickHouse。
   - **冷数据（30 天–1 年）**：对象存储（S3 / OSS），Parquet 格式，按天分区，Athena / Trino 查询。
4. **查询**：
   - 实时查询 → ES / ClickHouse。
   - 历史查询 → 异步查询服务，结果入库后推送通知。
5. **可靠性**：采集 → Kafka（at-least-once + idempotent producer）；Kafka → ES 消费者幂等（按 log_id）。

**容量估算（必须会算）**：
- 10GB/台/天 × 1000 台 = 10TB/天。
- 消息速率：假设日志行平均 500 字节 → 每天 2 × 10¹⁰ 行 → 每秒 ~23 万行（均分）→ 峰值可能 50 万行 QPS。
- Kafka 需要多少分区？按每个分区 10K 消息/秒算，需要 ≥50 个分区；保守 128 个。
- ES 集群规模：热数据 70TB（10TB × 7）× 1.5（副本+开销）≈ 105TB，按单节点 10TB 计算需 10+ 数据节点。

### 加分回答

- **采样 / 降级**：高 QPS 接口按比例采样；重要日志（ERROR）100% 保留。节省 50% 成本轻轻松松。
- **结构化日志标准**：全公司统一 JSON 格式 + 关键字段（trace_id, service, level, message），搜索体验天差地别。
- **告警联动**：ERROR 日志自增 → metrics → alerting；避免"看日志才发现的故障"。
- **数据生命周期**：ILM（Index Lifecycle Management）自动 rollover + 冷热迁移 + 删除。
- **成本对比**：ES 全量 30 天 vs. ES 7 天 + S3 + Athena 30 天，成本差 5~10 倍。
- **隐私**：采集前脱敏（mask 手机号 / 身份证）；sensitive topic 单独的 ACL。

### 常见失分

- 不做容量估算就设计。
- 只说"用 ELK"，不讨论成本和查询模式。
- 不处理采集端故障（断网 / 磁盘满）。
- 不考虑日志量峰值 vs. 均值的差异（大促期间可能 3x）。

## 场景 4 — 社交 Feed 系统

### 题面

> "设计一个朋友圈 / Twitter 式的 Feed 系统。用户关注其他用户，刷新首页时看到所有关注人按时间倒序的动态。活跃用户 1 亿，日均发布内容 5000 万条，首页刷新 QPS 峰值 50 万。要求首页加载 p95 < 300ms。"

### 考察重点

- **读写模式的权衡：推模式 vs. 拉模式 vs. 推拉结合**。
- **热点用户问题（明星效应）**。
- **数据模型 + 查询模式的对齐**。
- **大数据量下的排序和分页。**

### 需要澄清的问题

- 内容模型：只看关注人发的，还是还有推荐/广告？
- 动态是按绝对时间排序还是带算法排序？
- 未读 / 已读状态要维护吗？
- 删除动态的体验要求？（立刻消失 vs. 最终一致）
- 内容包含图片/视频吗？（影响存储和 CDN）
- 关注关系：单向 like Twitter 还是双向 like 朋友圈？

### 合格方案轮廓

**三种模式的对比**：

| | 推模式 (Fan-out on write) | 拉模式 (Fan-out on read) | 推拉结合 |
|---|---|---|---|
| 写操作 | 复制到所有粉丝收件箱 | 只写自己的发件箱 | 普通用户推 + 明星拉 |
| 读操作 | 从自己收件箱直接读 | 聚合所有关注人发件箱 | 收件箱 + 明星补拉 |
| 适合 | 关注数少、读远大于写 | 粉丝数悬殊、写放大严重 | 现实场景 |
| 劣势 | 明星 1 条帖子写 1 亿条 | 读高扇出，聚合慢 | 实现复杂 |

**推拉结合方案**：
1. **普通用户（粉丝 < 1 万）**：发布时 fanout 到所有粉丝的收件箱（Redis 有序集合，zset 按时间戳）。
2. **明星用户（粉丝 ≥ 1 万）**：发布时只写明星发件箱，不 fanout。
3. **读时**：
   - 拉用户自己的收件箱（普通用户的帖子已经在里面）。
   - 拉所有关注的明星的发件箱最新 N 条。
   - 内存中归并排序，取 top N。
4. **存储**：
   - 收件箱 / 发件箱：Redis 有序集合 + 过期 TTL（只保留最近 2000 条），超出的走 MySQL/HBase 翻页。
   - 帖子内容：NoSQL（HBase / Cassandra）按 user_id + ts 存储。
   - 关系：关注关系表（分库分表），缓存热用户关系链。

### 加分回答

- **一致性**：删除帖子的一致性如何保证？（收件箱里已经分发的副本需要清理或惰性过滤）
- **分页**：时间戳 + tiebreaker（postId）作为分页游标，避免偏移分页的性能坑。
- **下游服务分层**：feed 服务 / 关注服务 / 内容服务 / 推荐服务 清晰解耦。
- **推荐混排**：算法排序时，时间序只是一维；实际应该是 rank 服务输出候选 + 多维度排序。
- **冷启动 / 新用户**：收件箱空 → 走推荐 pool。
- **故障降级**：明星服务挂了，降级到只看已在收件箱的内容。
- **容量估算**：
  - 写 QPS：5000 万 / 86400 ≈ 580 QPS 均值，峰值可能 5000。
  - 读 QPS：50 万（题目给定）。
  - fanout 写放大：假设平均每用户 500 粉丝，fanout 后写操作 = 580 × 500 = 29 万/秒 → Redis 集群需要足够分片。

### 常见失分

- 一上来就选推模式，不考虑明星问题。
- 用关系型 JOIN 实现 feed 聚合（刷一次首页 JOIN 100 条 tweet，数据库会死）。
- 不考虑缓存过期 / 失效策略。
- 不估算容量，吹牛皮"用 Redis 撑 50 万 QPS"没论证。

## 场景 5 — 分布式 ID 生成器

### 题面

> "公司从单体拆微服务后，订单 / 商品 / 用户各自需要全局唯一 ID。要求：单机 1 万 QPS 生成 ID；ID 趋势递增（便于 DB 索引）；不同机房和机器都能独立生成无冲突；ID 长度不宜过长（不能用 UUID 32 位）。"

### 考察重点

- **常见 ID 方案的对比和权衡**。
- **时钟回拨的处理**（Snowflake 的经典坑）。
- **机器号分配的管理复杂度**。
- **数字的工程直觉**：64 位能表示多久？

### 需要澄清的问题

- 有多严格的"全局唯一"？跨机房 + 跨服务？
- "趋势递增"能接受多大抖动？严格递增 vs. 粗粒度递增？
- 机器规模？会动态扩缩容吗？
- 对 ID 的隐私要求？（猜下一个 ID 敏感吗？订单 ID 通常要不可枚举）
- 历史数据是否需要迁移到新 ID 体系？

### 方案对比

| 方案 | 优点 | 缺点 |
|---|---|---|
| **UUID** | 无中心化，实现简单 | 36 字符太长，无序（B+ 树索引插入热点切换） |
| **DB 自增 + 号段** | 趋势递增，实现直观 | DB 成为瓶颈；号段分配失败场景复杂 |
| **Snowflake** | 64 位，有时序，高性能 | 依赖时钟，回拨风险；机器号管理 |
| **Leaf-Segment** | 基于 DB 号段 + 批量预取 | 依赖 DB |
| **Leaf-Snowflake** | Zookeeper 管理 workerId | ZK 依赖 |
| **UUIDv7** | 时序 + 无中心 | 长度仍是 36 |

### 合格方案（推荐 Snowflake 或变种）

**Snowflake 基本结构（64 位）**：
- 1 位符号（保留 0）
- 41 位时间戳（ms 级，可用约 69 年）
- 10 位机器号（1024 台机器）
- 12 位序列号（单机单 ms 4096 个 ID）

**关键问题与解**：
1. **时钟回拨**：
   - 启动时检测：上一次生成的时间 > 当前时间 → 抛异常，拒绝启动。
   - 运行时回拨：
     - 小幅（< N ms）：等待直到回拨结束。
     - 大幅：报警 + 停止发号；或使用扩展位做"回拨计数"隔离。
2. **机器号分配**：
   - ZooKeeper / etcd 持久顺序节点。
   - 或本地配置 + CMDB 记录，启动时校验唯一性。
3. **并发**：单机单 ms 4096 不够时的溢出 → 等下一毫秒。

### 加分回答

- **生产经验**：讲一次机器号冲突导致的事故，或时钟同步出问题的教训。
- **业务友好变种**：前缀 + ts + 机器 + 自增的人类可读形式（例如 `ORD202604230001`）。
- **隐私考虑**：订单 ID 不应可枚举，可加入噪声位或 AES 加密后对外；内部仍用顺序 ID 做索引。
- **业务/存储关联**：ID 里嵌业务信息（分片键 / 分库提示），但要小心强耦合。
- **对比真实工业方案**：百度 UidGenerator（改进版 Snowflake，时间回拨容忍）、美团 Leaf、Twitter 原生 Snowflake 的差异。

### 常见失分

- 把 UUID 当默认答案，不问需求。
- 不提时钟回拨。
- 机器号靠"配置文件写死"，扩容时容易冲突。
- 没有回答单机 QPS 测算（每毫秒 4096 = 单机 400 万 QPS，远超题目要求）。
